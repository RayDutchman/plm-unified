# 主页仪表盘重构设计

## 1. 需求背景

现有主页仪表盘（`frontend/src/pages/Dashboard.tsx`）过于单调：4 张外观完全一致的统计卡片（零件 / 部件 / 图文档 / 用户），每张都是同一个 5 格数字网格，纯垂直堆叠，再加一排快捷操作。存在两个核心问题：

1. **视觉单调**——尺寸、类型、节奏高度雷同，缺乏层次。
2. **内容偏静态、偏管理视角**——只有全局计数，用户登录后看不到"和我有关、需要我处理"的东西，无法"接着干活"。

本次重构目标：将仪表盘改造成**"个人工作区 + 全局概览"两者融合**的面板，既能让用户第一眼看到待办并直接点进去处理，又能一览系统全局。

## 2. 设计决策

经脑暴确认以下关键决策：

| 决策项 | 结论 |
| --- | --- |
| 核心定位 | 两者融合：上半屏个人工作区，下半屏全局概览 |
| 整体骨架 | 混合式（分区骨架 + 区内错落磁贴）：两个区各为独立网格，区内用大小不一的 bento 磁贴打破单调 |
| 个人区内容 | 待我处理（主角）、我的任务、最近访问、我的收藏 |
| "最近访问"语义 | 分两期：一期用"最近编辑"（日志派生，零埋点）；二期加详情页 localStorage 埋点，升级为真浏览历史 |
| 全局区图表 | 状态分布用**分段条**（同时对比三类实体的状态构成），非环图 |
| 数据模型 | 不新增数据库表/字段；"待我处理""我的任务"因列表 brief 不含评审人/无跨项目任务接口，各需新增一个后端只读聚合接口 |
| 待我处理覆盖范围 | 一期覆盖 ECR + ECO（待我审批 + 我发起被驳回）；构型/物料审批并入二期 |

## 3. 布局结构

沿用两区分层，每区是一个独立的 4 列（桌面）网格，可各自响应式塌陷、各自处理空态。

```
┌─ 问候抬头：早上好，<姓名> · 你有 N 项待处理、M 个任务逾期 ──────────┐
│                                                                      │
│  个人工作区（accent 边框 / 浅色背景，视觉更"暖"）                    │
│  ┌───────────────┬───────────────────────────┐                      │
│  │               │  我的任务（宽磁贴，逾期红标）│                     │
│  │  待我处理      ├─────────────┬─────────────┤                      │
│  │  （2×2 主磁贴）│  最近访问    │  我的收藏    │                     │
│  │               │  （小方贴）  │  （小方贴）  │                     │
│  └───────────────┴─────────────┴─────────────┘                      │
│  ───────────────────────── 分隔线 ─────────────────────────         │
│  全局概览（中性色）                                                   │
│  ┌────┬────┬────┬────┐  4 个 KPI 小卡                               │
│  │零件│部件│文档│变更│                                              │
│  └────┴────┴────┴────┘                                              │
│  ┌───────────────┬───────────────┐                                  │
│  │  状态分布      │  系统动态流     │  两个 2×2 大磁贴                 │
│  │  （分段条）    │  （活动流）     │                                 │
│  └───────────────┴───────────────┘                                  │
└──────────────────────────────────────────────────────────────────────┘
```

### 响应式

- 桌面（宽）：如上，个人区 2 列（主磁贴 | 右侧组），全局区 KPI 4 列、图表 2 列。
- 窄屏：个人区塌为单列（待我处理全宽 → 我的任务 → 最近访问 + 我的收藏两列或堆叠）；全局区 KPI 变 2 列、两个大磁贴各自全宽。
- 两区是独立网格，塌陷互不牵连。

### 空态

每块磁贴自带友好空态（如"✅ 暂无待办""还没有收藏，去各页面添加"），任一数据源缺失或离线只显示该磁贴空态，不影响其它磁贴，也不撕出布局空洞。

## 4. 各磁贴内容与数据来源

### 4.1 问候抬头

- 内容：`早上好/下午好/晚上好，<real_name> · 你有 N 项待处理、M 个任务逾期`。
- 数据：当前登录用户（`useAuthStore`）+ 待我处理计数 + 逾期任务计数。
- 目的：利用现有空白顶部，提供个性化入口与当日聚焦。

### 4.2 待我处理（主磁贴 2×2）

聚合我需要行动的事项，按时间/优先级排序，磁贴内直接内嵌前 4~5 条，底部"查看全部 N 项 →"跳转。每条：类型标签（ECR/ECO/构型/物料）+ 编号 + 标题 + 优先级点/状态 + 相对时间。

数据来源（我 = 当前 `user_id`）。**列表接口的 brief 响应不含 `reviewers`**，前端无法据此判断"待我审批"，故由后端新增只读聚合接口 `GET /dashboard/my-todos` 统一计算并返回：

| 类型 | 判定条件（后端） | 期次 |
| --- | --- | --- |
| ECR 待我审批 | `status=reviewing` 且 `reviewers` JSONB 含我 且我在 `ecr_review_records` 无记录 | 一期 |
| ECO 待我审批 | 同上（ECO 对应表） | 一期 |
| 我发起被驳回 | `creator_id==我` 且 `status=rejected` | 一期 |
| 构型/物料待我审批 | 同模式（`ProfileReviewer` / `InvReviewer`） | 二期 |

接口返回统一结构 `{ type, kind, id, number, title, priority, status, updated_at }`，前端按 `updated_at` 倒序展示，磁贴内取前 4~5 条。

### 4.3 我的任务（宽磁贴）

- 内容：指派给我且未完成的项目任务，前若干条，逾期条目标红（"逾期 N 天"）。
- 数据：`ProjectTask` 中 `assignee_id==我` 且 `status != 已完成`；逾期 = `planned_end` 早于今天。
- 接口：现有 `projectApi` 只能按单个项目 `listTasks(projectId)`，无跨项目"我的任务"接口，故后端新增只读聚合接口 `GET /projects/my-tasks`，返回 `{ project_id, project_name, task_id, code, name, status, priority, planned_end }`。

### 4.4 最近访问（小方贴）

- **一期**：显示"我最近编辑/创建"的实体。数据 = 操作日志 `logsApi.list({ user_id: 我, action: create|update })` 去重取最近 N 条，展示类型图标 + 编号 + 名称。
- **二期**：在零件/部件/图文档/构型详情页打开时向 localStorage 写入一条访问记录（含类型、id、编号、名称、时间戳），仪表盘读取本地记录，升级为真实浏览历史。二期为独立增量，不阻塞一期上线。

### 4.5 我的收藏（小方贴）

- 内容：个人工作台收藏/归档的常用项，图标 + 编号 + 名称，点击跳转。
- 数据：Board 个人收藏项（`boardApi`）。

### 4.6 全局 KPI（4 小卡）

- 零件总数、部件总数、图文档总数、变更进行中。
- 前三项复用现有本地缓存计数（`useDataStore` + `countByStatus`），零成本。
- "变更进行中" = ECR/ECO 中 `status=reviewing` 计数（list count 或 `dashboardApi.getStats`）。

### 4.7 状态分布（大磁贴 2×2）

- 内容：零件 / 部件 / 图文档三行，各一条水平分段条，分段 = 草稿 / 冻结 / 发布 / 作废，底部图例。
- 颜色：草稿=蓝、冻结=橙、发布=绿、作废=红（沿用现有 `STATUS_COLORS` 语义）。
- 数据：本地缓存 `countByStatus` 结果，零成本。
- 选型理由：分段条能在一屏内横向对比三类实体的状态构成，信息密度优于三个独立环图。

### 4.8 系统动态流（大磁贴 2×2）

- 内容：最近的系统操作记录，每条 = 用户首字头像 + "谁 做了什么 于 哪个对象" + 相对时间，对象编号可点击跳转。
- 数据：`logsApi.list`（全局，最近 N 条）。

## 5. 组件结构

在 `frontend/src/pages/Dashboard.tsx` 内按磁贴拆分为聚焦子组件，各自单一职责、可独立理解与测试；数据获取用各自的 hook 封装，失败静默降级为空态：

- `GreetingHeader`
- `MyTodosTile`（待我处理）
- `MyTasksTile`（我的任务）
- `RecentItemsTile`（最近访问）
- `FavoritesTile`（我的收藏）
- `KpiStrip`（4 KPI）
- `StatusDistributionTile`（状态分布分段条）
- `ActivityFeedTile`（系统动态流）

现有 `StatusCard` / `RoleCard` / `QuickAction` 视情况保留复用或替换。若 `Dashboard.tsx` 增长过大，将子组件与 hooks 拆到 `frontend/src/pages/Dashboard/` 目录下独立文件。

## 6. 分期

- **一期（MVP）**：完整两区布局 + 全部磁贴。含两个新后端聚合接口（`GET /dashboard/my-todos` 覆盖 ECR/ECO、`GET /projects/my-tasks`）。"最近访问"用"最近编辑"语义。
- **二期（增量）**：详情页 localStorage 浏览埋点，"最近访问"升级为真浏览历史；`my-todos` 扩展覆盖构型/物料审批。

## 7. 不做（Out of Scope）

- 不新增数据库表/字段。
- 不做仪表盘布局的用户自定义拖拽（未来可选，本次不做）。
- 不改动各业务页面自身逻辑（除二期详情页埋点外）。
- 不引入重型图表库；分段条/KPI 用原生 div + Tailwind 实现，保持轻量。
